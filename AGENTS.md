## What this project is

A static, single-page web app that generates printable letter/number/word practice sheets for pre-K and kindergarten kids. Output quality matters; simplicity of the implementation is the second priority.

Primary user: a parent configuring sheets for their child. Secondary: anyone who opens a shared URL.

## Architecture at a glance

Single source of truth: `Store<SheetConfig>` (in `src/state/store.ts`).

```
┌──────────────┐   update   ┌──────────────────────┐   subscribe   ┌────────────────┐
│     form     │ ─────────▶ │                      │ ─────────────▶│ URL (history)  │
│  (radios,    │            │  Store<SheetConfig>  │               └────────────────┘
│   textarea)  │ ◀───────── │                      │ ─────────────▶ ┌────────────────┐
└──────────────┘  subscribe └──────────────────────┘                │ preview (DOM)  │
                                         ▲                          └────────────────┘
                          popstate event │
                          (back/forward)

```

Rendering is all SVG, composed from font glyph paths extracted by `opentype.js`. No `<canvas>`, no image assets at runtime (only the Andika TTF).

## Invariants — don't break these

1. **Ruled-line geometry derives from font metrics.** `computeLines(asset, capHeightPx)` in `src/rendering/ruled-lines.ts` produces all y-coordinates from the loaded font's metrics. Never hardcode y-offsets for ruled lines.

2. **Physical units for print.** Use `in` / `mm` / `px` at 96dpi (where 1in = 96 CSS px). Don't mix units such that print output differs from on-screen dimensions.

3. **URL reflects state.** Any change to `SheetConfig` must be mirrored in the URL (see `configToURLParams`). Any new field in `SheetConfig` requires updating both `configFromURL` and `configToURLParams`.

4. **Auto-generated files are not hand-edited.** `src/rendering/skeletons/<font>-baseline.ts` is generated by `scripts/generate-skeletons.ts`. To change skeleton data, change the generator's parameters or inputs and rerun `npm run generate:skeletons:<font>` (e.g. `:andika`).

5. **TDD for pure logic.** Content parsing, URL codec, font-metric math, and state management all have unit tests. Preserve that coverage when modifying.

## Conventions

- **TypeScript strict mode.** `noUncheckedIndexedAccess` is on — be explicit when indexing arrays.
- **No framework.** Plain DOM. Don't pull in React, Solid, etc. unless you've discussed first. Current UI complexity doesn't warrant it.
- **No external CSS frameworks.** Plain CSS in `src/styles/`.
- **Commit per logical step.** Multi-step changes commit at each working milestone, not one big commit at the end.
- **Visual regressions require intent.** After changing anything in `src/layout/` or `src/theming/`, run `npm run test:e2e`. If snapshots changed by intent, regenerate with `npm run test:e2e:update` and include them in the commit.

## Where things live

See `README.md` → "Development → Project layout" for the canonical map.

Highlights to remember:
- Types first: `src/config/types.ts` defines `SheetConfig` and its constituent unions.
- URL round-trip: `src/config/url.ts`. Newline-separated content; `\n` → `%0A`.
- Row composition: `src/layout/row.ts`. Each row draws ruled lines + word-slots; each slot is solid demo, dashed trace (skeleton + ghost fill), or blank.
- Sheet composition: `src/layout/sheet.ts`. Each line in `config.content` becomes one row; rows are packed into pages top-to-bottom.
- Starters: `src/config/presets.ts`. A typed list of `{key, label, config: SheetConfig}` entries; the form dropdown maps selection → `store.set(preset.config)`.

## Common traps

- **Stroke width under transforms.** Skeleton paths are transformed with `scale(s, -s)` to map font units to SVG pixels. Stroke-width and stroke-dasharray would be scaled too; avoid by always using `vector-effect: non-scaling-stroke` on stroked skeleton paths.
- **SVG viewBox must include ascender overflow.** Letters like `f`, `h`, `b`, `l`, `t` reach above cap-height in Andika. `computeLines()` already accounts for this via `ascenderLine` at y=0 and `headline` below it; don't bypass it.
- **Row geometry uses measured letter bounds, not font metrics.** `FontAsset.ascender`/`descender` are the tightest box over A–Z/a–z/0–9 (computed at load time in `font.ts`) — not the font's linguistic `ascender`/`descender` values, which for Andika include diacritic room we never use.
- **Empty content lines are meaningful.** Blank lines produce blank ruled-line rows. Trailing blank lines are preserved. Don't strip them in `parseContent`.
- **Motifs must fit the margin strip.** `placement.ts` caps rendered motif size so that at ±12° rotation the bounding box stays within `marginPx`. Raising `BASE_MOTIF_PX` or max `scale` without matching the cap will push motifs off the page or into content.

## Testing quick reference

```bash
npm test                 # Vitest, ~1s
npm run test:e2e         # Playwright, builds + runs against dist, ~15s
npm run test:e2e:update  # Regenerate 15 snapshots
npm run generate:skeletons:andika         # Regenerate Andika centerline data
npm run generate:skeletons:comic-relief   # Regenerate Comic Relief centerline data (inspector only)
npx tsc -b --noEmit      # Type check without build
```

Smoke test when adding a feature: build (`npm run build`), start dev server, click through the controls, confirm URL updates and preview re-renders, open print preview.
